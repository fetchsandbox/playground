# surge — 2026-06-25

**Prompt:** `./fetchsandbox we got a TCPA complaint about texting opted-out customers — investigate the surge integration`

**What happened:**

Brain routed to the `surge` spec at 0.6 confidence (no Tier-1 workflow matched the TCPA intent
out of the box). Imported the OpenAPI spec generated from the FastAPI app, ran
`contacts_create` + `surge_webhook_create` with a `contact_opted_out` scenario.

Two bugs found in `main.py`:

1. **Opt-out webhook silently dropped** — the `surge_webhook` handler only branched on
   `message.delivered`. Events of type `contact.opted_out` / `message.opted_out` fell through
   without updating `sms_status`, leaving every opted-out contact permanently marked `"active"`.

2. **No pre-send opt-out guard** — `send_reminder` called the Surge API unconditionally.
   Even if `sms_status` had been set correctly by the webhook, nothing prevented the SMS
   from being dispatched.

Together these mean any customer who replies STOP keeps receiving appointment reminders —
a clear TCPA violation.

**Fix applied:**
- Webhook handler now handles `contact.opted_out` / `message.opted_out` → sets
  `sms_status = "opted_out"` on the matching contact.
- `send_reminder` now raises HTTP 403 before touching Surge if `sms_status == "opted_out"`.

The sandbox reproduced the before state (opt-out webhook returns 200 but no state change)
and confirmed the after state passes. Brain was useful for routing to the right spec; the
missing piece was a curated TCPA/opt-out workflow — worth adding as Tier-1 for this spec.
Score: 7/10.
